## 基本参数：

### `tuner_backend`

在 ms-swift（ModelScope Swift）框架中，tuner_backend 是一个用于指定微调（fine-tuning）后端实现的命令行参数。该参数决定了在进行参数高效微调（Parameter-Efficient Fine-Tuning, PEFT）时，底层使用哪个库或技术栈来实现具体的微调方法（如 LoRA、Adapter 等）。

**参数说明：**

- **参数名**：`-tuner_backend`

- **可选值**：
    
    - `'peft'`（默认）
    
    - `'unsloth'`
    

- **默认值**：`'peft'`

**详细解释：**

- `**peft**`**（默认后端）**
    
    - **来源**：Hugging Face 官方维护的 [PEFT](https://github.com/huggingface/peft) 库。
    
    - **特点**：
        
        - 支持多种 PEFT 方法，如 LoRA、Prefix Tuning、Prompt Tuning、IA³、Adapter 等。
        
        - 与 Hugging Face Transformers 无缝集成。
        
        - 社区广泛使用，文档完善，稳定性高。
        
        - 支持大多数主流模型架构。
        
    
    - **适用场景**：通用微调任务，尤其适合需要多种 PEFT 方法支持或与 Hugging Face 生态深度集成的项目。
    

- `**unsloth**`
    
    - **来源**：[Unsloth](https://github.com/unslothai/unsloth) 是一个专注于**加速 LoRA 微调**的第三方库。
    
    - **特点**：
        
        - 专为 LoRA 优化，通过内核融合（kernel fusion）、内存优化等技术显著提升训练速度（官方宣称可提速 2–5 倍）并降低显存占用。
        
        - 仅支持 LoRA（目前不支持其他 PEFT 方法）。
        
        - 与 Hugging Face 模型兼容，但需额外安装 unsloth 库。
        
        - 更适合资源受限或追求训练效率的场景。
        
    
    - **适用场景**：当你**只使用 LoRA** 进行微调，并希望获得更快的训练速度和更低的显存消耗时，推荐使用 `unsloth` 后端。
    

**总结对比：**

|特性|PEFT|UNSLOTH|
|---|---|---|
|支持的 PEFT 方法|多种（LoRA、Adapter 等）|仅 LoRA|
|训练速度|标准|显著更快（2–5x）|
|显存占用|较高|更低|
|兼容性|广泛兼容 Hugging Face|兼容但需额外安装|
|默认选项|✅ 是|❌ 否|

### `train_type`

在 **ms-swift**（ModelScope Swift）框架中，`train_type` 是一个关键的命令行参数，用于指定模型微调（fine-tuning）时采用的具体**训练策略或参数高效微调（PEFT）方法**。不同的 `train_type` 对应不同的参数更新机制、计算开销、显存占用和适用场景。

**参数说明：**

- **参数名**：`-train_type`

- **可选值**：
    
    - `'lora'`（默认）
    
    - `'full'`
    
    - `'longlora'`
    
    - `'adalora'`
    
    - `'llamapro'`
    
    - `'adapter'`
    
    - `'vera'`
    
    - `'boft'`
    
    - `'fourierft'`
    
    - `'reft'`
    

- **默认值**：`'lora'`

**各选项详细解释：**

### **1.** `**'lora'**`**（Low-Rank Adaptation）**

- **原理**：在原始权重矩阵旁路添加低秩分解矩阵（A×B），仅训练这两个小矩阵，冻结主干模型。

- **优点**：
    
    - 显存占用低，训练快。
    
    - 广泛支持各类模型（LLaMA、Qwen、ChatGLM 等）。
    
    - 与 Hugging Face PEFT/Unsloth 兼容。
    

- **适用场景**：通用高效微调，尤其适合资源有限的场景。

- **默认选项**，社区最成熟。

### **2.** `**'full'**`**（Full Fine-Tuning）**

- **原理**：对整个模型的所有参数进行端到端训练（不冻结任何层）。

- **优点**：
    
    - 理论上性能上限最高。
    
    - 无需特殊结构修改。
    

- **缺点**：
    
    - 显存和计算开销极大（通常需要多卡或模型并行）。
    
    - 容易过拟合小数据集。
    

- **适用场景**：数据量大、算力充足、追求极致性能。

### **3.** `**'longlora'**`

- **原理**：基于 LoRA 的扩展，专为**扩展上下文长度**（如从 2K → 32K tokens）设计。

- **关键技术**：
    
    - 结合 **Shifted Sparse Attention** 和 LoRA。
    
    - 在不重新训练位置编码的情况下扩展上下文。
    

- **适用场景**：需要处理超长文本（如长文档、代码、书籍）的微调任务。

- **注意**：通常需配合支持长上下文的模型（如 LLaMA-2-7B-32K）。

### **4.** `**'adalora'**`**（Adaptive LoRA）**

- **原理**：LoRA 的改进版，动态调整各层 LoRA 矩阵的秩（rank），自动分配参数预算。

- **优点**：
    
    - 更高效利用参数预算。
    
    - 在相同参数量下通常优于标准 LoRA。
    

- **来源**：论文 _AdaLoRA: Adaptive Budget Allocation for Parameter-Efficient Fine-Tuning_。

- **适用场景**：希望在有限参数预算下获得更好性能。

### **5.** `**'llamapro'**`

- **原理**：针对 LLaMA 架构提出的结构化扩展方法，**增加额外的专家层（MoE-like）或中间层**，同时保留原始权重。

- **特点**：
    
    - 不是传统 PEFT，而是“模型扩展 + 微调”。
    
    - 可显著提升模型容量和表达能力。
    

- **适用场景**：希望在 LLaMA 系列模型上做能力增强（如增加推理深度），且能接受稍高的显存开销。

### **6.** `**'adapter'**`

- **原理**：在 Transformer 层中插入小型前馈网络（Adapter 模块），仅训练这些模块。

- **结构**：通常为“降维 → 非线性 → 升维”瓶颈结构。

- **优点**：
    
    - 模块化设计，易于插入。
    

- **缺点**：
    
    - 推理时有额外计算延迟（需过 Adapter）。
    
    - 相比 LoRA，参数效率略低。
    

- **适用场景**：早期 PEFT 方法，适合研究对比或特定架构适配。

### **7.** `**'vera'**`**（VeRA: Variance Stabilized Random Projection）**

- **原理**：固定随机投影矩阵 P 和 Q，仅训练对角缩放向量 d 和 b，实现超低参数微调。

- **优点**：
    
    - 参数量极小（仅两个向量）。
    
    - 训练更稳定（方差控制）。
    

- **来源**：2024 年新方法（_VeRA: Parameter-Efficient Finetuning with Fixed Random Projections_）。

- **适用场景**：极端参数受限场景（如 < 0.1% 可训练参数）。

### **8.** `**'boft'**`**（Blockwise Orthogonal Finetuning）**

- **原理**：通过**分块正交变换**（Block-wise Orthogonal Transform）更新权重，保持正交性以提升优化稳定性。

- **优点**：
    
    - 理论上优化更稳定。
    
    - 适用于深层模型。
    

- **来源**：ICLR 2024 论文。

- **适用场景**：研究前沿方法，或需要高训练稳定性的任务。

### **9.** `**'fourierft'**`**（Fourier Transform-based Fine-Tuning）**

- **原理**：在频域（Fourier 域）对权重进行稀疏更新，利用频域稀疏性减少参数。

- **特点**：
    
    - 利用信号处理思想进行参数压缩。
    
    - 较新方法，仍在探索阶段。
    

- **适用场景**：实验性研究，或特定频域敏感任务。

### **10.** `**'reft'**`**（Representation Finetuning）**

- **原理**：不修改模型权重，而是**直接干预中间层的隐藏表示（activations）**，通过学习一个干预函数（如低秩映射）来调整表示。

- **代表方法**：LoReFT、NITRO 等。

- **优点**：
    
    - 可实现“零权重修改”微调。
    
    - 支持非常灵活的干预策略。
    

- **适用场景**：可控生成、表示编辑、轻量级任务适配。

### **对比总结表**

||||||
|---|---|---|---|---|
|`full`|100%|极高|✅|性能上限高，资源消耗大|
|`lora`|~0.1–1%|低|✅✅✅|默认，高效稳定|
|`adalora`|~0.1–1%|低|✅|动态分配秩，更高效|
|`longlora`|~0.1–1%|中|✅|支持超长上下文|
|`llamapro`|~5–20%|中高|⚠️|扩展模型结构，非纯 PEFT|
|`adapter`|~1–3%|中|✅|早期 PEFT，有推理延迟|
|`vera`|<<0.1%|极低|🔬|超低参数，固定投影|
|`boft`|~0.5–2%|中|🔬|正交约束，优化稳定|
|`fourierft`|~0.1–1%|低|🔬|频域稀疏，较新|
|`reft`|~0.1%|低|🔬|干预表示，不改权重|

> ✅：成熟可用；🔬：研究前沿/实验性；⚠️：特殊用途

### **使用建议**

- **新手/通用场景**：使用默认 `lora`。

- **追求极致性能 & 有足够资源**：尝试 `full`。

- **需要处理长文本**：选 `longlora`。

- **想尝试最新高效方法**：可试 `adalora` 或 `vera`。

- **研究目的**：可探索 `boft`、`fourierft`、`reft` 等。

> 注意：部分方法（如 llamapro、longlora）可能对模型架构有特定要求（如仅支持 LLaMA 系列），使用前请查阅 ms-swift 官方文档或示例。
